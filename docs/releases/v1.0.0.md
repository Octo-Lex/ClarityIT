**Release Date**: 2026-06-14
**Codename**: Sovereign Operations

## Overview

ClarityIT v1.0.0 is the first production-ready release of the AI-native IT operations platform. It delivers identity-driven access, policy-governed agent autonomy, controlled Proxmox infrastructure mutations, operator-controlled remediation, and comprehensive security controls — all designed for sovereign (self-hosted) deployment.

## What's New Since v0.9.0

### Real MFA (Track 1)
- RFC 6238 TOTP authentication with AES-256-GCM encrypted secret storage
- HKDF-SHA256 key derivation from dedicated `MFA_KEY`
- 10 single-use HMAC-SHA256 hashed recovery codes
- `RequireRecentMFA` middleware for high-risk action gating
- 5-minute MFA validity window, 5-attempt lockout with 15-minute cooldown

### Approval Workflow Engine (Track 2)
- Risk-based approval policies (low auto-approve → critical 2 approvers + MFA)
- 7-state lifecycle: pending → {approved, rejected, cancelled, expired} → {executed, failed}
- Immutable decisions with UNIQUE(approval_id, decided_by)
- Self-approval prevention
- Sanitized action_target storage (15+ sensitive key patterns redacted)

### Policy-Governed A4 Execution (Track 3)
- 13-check PolicyEvaluator enforcement chain in exact order
- A5 autonomy hardcoded rejection (fails before any DB lookup)
- 10-point approval verification (action_type match, target deep-equal, expiry, cross-team, executed_at)
- Full tool gateway integration with audit/outbox/effect results

### Proxmox Controlled Mutation Pipeline (Track 4)
- 4 allowed mutation actions: start, shutdown, stop, snapshot
- Risk policy: start/snapshot=medium, shutdown=high+MFA, stop=critical+2approvers+MFA
- `PROXMOX_MUTATION_ENABLED` feature flag (default: false)
- Snapshot name regex validation (shell injection prevention)
- 16 execution guardrails

### Operator-Controlled Remediation (Track 5)
- Proposal lifecycle: draft → proposed → approved → executing → {completed, failed, cancelled}
- Agent proposals always start as draft
- Step execution through Tool Gateway (full 13-check chain)
- `continue_on_failure` for partial success
- Auto-creates "remediation-executor" system agent per team

### Operator UI (Track 6)
- 5 new frontend pages: Security (MFA enrollment), Approvals, Asset Actions, Asset Action Requests, Remediation Panel
- Permission-gated routes with PermissionRoute component
- Risk indicators (medium/high/critical color coding)
- Recovery codes shown once with explicit save warning
- Sensitive data redaction in UI matching backend

### Security Review Closure (Track 7)
- 5 security documents (threat model, high-risk actions, MFA/approval model, Proxmox boundary, agent autonomy)
- 16 security tests (secret leakage, forbidden routes, A5 disabled, MFA window, sanitization, immutability)
- Recovery code hash fixed to HMAC-SHA256
- Web container hardened (nginx-unprivileged, read-only filesystem, non-root)
- Production dependency audit clean (dev-only findings formally risk-accepted)

## Security Posture

- All high-risk actions require: MFA + approval + policy + audit + outbox
- A5 autonomy disabled (hardcoded, pre-DB rejection)
- No destructive mutations (delete/migrate/clone/firewall/network/storage/cert)
- Python reasoning workers have zero infrastructure access
- Production dependency audit: 0 vulnerabilities
- Non-root containers with read-only filesystems

## Deployment

Sovereign Hybrid: Proxmox LXC + Cloudflare Tunnel

```bash
# Clone and configure
git clone <repo> /opt/clarityit
cd /opt/clarityit
cp services/api/.env.example .env  # Edit with production values

# Deploy
docker compose up -d --build

# Verify
make verify-deployment
```

## Known Limitations

1. Single-tenant deployment (one organization per instance)
2. Proxmox mutation feature-flagged off by default
3. Dev-only npm audit findings (esbuild/vite chain) — formally risk-accepted, not in production image
4. Email delivery requires SMTP configuration (no built-in email server)
5. No automated migration rollback (manual SQL required)
