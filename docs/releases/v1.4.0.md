## Overview

v1.4.0 introduces **ClarityDocs** — ClarityIT's native document productivity system. ClarityDocs is built on a PostgreSQL JSONB document model with a block-based editor, agent-assisted editing, structured document generation, reusable templates, multi-format export, and non-destructive version history.

**ClarityIT owns the document model. The agent helps draft. The user applies edits. The Go API stores everything.**

## Tracks

### Track 1 — Native Document Artifact Model
- `artifact_documents` table with `document_json` JSONB column
- 8 block types: heading, paragraph, bullets, numbered_list, table, quote, callout, page_break
- 10 document types: general_document, decision_memo, implementation_plan, incident_summary, training_doc, architecture_doc, project_report, status_report, meeting_summary, executive_brief
- Schema version 1 enforced at API and DB level
- Server-side word count computation
- Migration 036

### Track 2 — ClarityDocs Editor UI
- `DocumentEditorPage` with block-based editor
- `DocumentBlockEditor` for per-block editing (text, items, table cells, variants)
- `DocumentOutline` for navigation
- `DocumentToolbar` for adding blocks
- `DocumentSaveStatus` for save state feedback
- Dirty state tracking, auto-save on manual trigger

### Track 3 — Agent Assist for Documents
- `POST /api/teams/{teamId}/artifacts/documents/{artifactId}/document-assist`
- 10 assist modes: rewrite, summarize, expand, make_concise, make_executive, make_technical, draft_section, create_outline, extract_action_items, improve_headings
- Python reasoning worker `StubDocumentAssistGateway` on port 9100
- Suggestions are preview-only — user applies via Insert/Replace
- **No DB writes, no autosave, no MinIO access from worker**
- `chain_of_thought` rejected; `reasoning_summary` only
- Worker endpoint internal-only (Docker network, not published to host)

### Track 4 — Internal Document Generation
- `POST /api/teams/{teamId}/artifacts/generate-document`
- Validates: title ≤200, document_type allowlist, prompt ≤2000, tone (technical/executive/casual/formal), sections ≤20
- Python `StubDocumentGenerateGateway` returns structured document_json
- Go API validates blocks, recomputes word_count, persists transactionally
- `source_data` stores metadata but NOT raw prompt
- `DocumentGenerateModal` frontend with type/tone/section selectors

### Track 5 — Native Document Templates
- Migration 037: `template_format`, `document_json`, `schema_version` columns on `artifact_templates`
- 7 system document_json templates + 6 existing markdown templates
- 4 DB-level CHECK constraints including `atpl_schema_version_by_format`
- `InstantiateTemplate` creates artifact + artifact_documents for document_json templates
- `TemplateGallery` with format badges, DocBlockPreview, custom template creation

### Track 6 — DOCX/PDF/Markdown Export
- `GET /api/teams/{teamId}/artifacts/{artifactId}/export/markdown`
- `GET /api/teams/{teamId}/artifacts/{artifactId}/export/pdf`
- `GET /api/teams/{teamId}/artifacts/{artifactId}/export/docx`
- **Pure Go implementations** — no SuperDoc, no LibreOffice, no external document engine
- DOCX: OOXML writer using `archive/zip`, all user content XML-escaped
- PDF: Go PDF 1.4 format (v1.3 renderer extended for native documents)
- Markdown: `document_json` → markdown rendering for all 8 block types
- **Streaming-only** — no MinIO storage writes, `last_exported_storage_object_id` intentionally unchanged
- Non-document artifacts fall back to v1.3 `content_markdown` export

### Track 7 — Version History

- `source` field: user_save, agent_assisted_edit, generated, template, restore
- Non-destructive restore: creates new version with old content
- `GET /versions`, `GET /versions/{versionId}`, `POST /versions/{versionId}/restore`
- Save conflict detection (Option A): `If-Match` header → 409 on stale save

## Architecture

### ClarityDocs Data Flow

```
PostgreSQL JSONB (source of truth)
  └── artifact_documents.document_json
       ├── schema_version: 1
       ├── title, document_type
       └── blocks: [{id, type, level?, text?, items?, headers?, rows?, variant?}]

Exports (generated on-demand, never persisted):
  ├── document_json → Markdown (renderBlocksToMarkdown)
  ├── document_json → Markdown → PDF (buildMinimalPDF)
  └── document_json → DOCX (buildDOCX, pure Go OOXML)

Version History (append-only):
  └── artifact_document_versions (full JSONB snapshots)
       └── Restore: non-destructive, creates new version
```

### Key Design Decisions

- **No SuperDoc dependency** — ClarityIT owns the document model entirely
- **No copied SuperDoc code** — all code is original
- **No external document engine** — no LibreOffice, no browser-only export
- **PostgreSQL JSONB as source of truth** — DOCX is export-only format
- **Python worker isolation** — zero DB/MinIO/NATS/Redis access
- **Non-destructive everything** — versions, restore, archive (DELETE = archive)

## Security Boundaries

- No public document sharing
- No approval workflow for documents
- No Tool Gateway execution for documents
- No Proxmox mutation changes
- No A5/autonomy expansion
- No Python DB/MinIO/NATS/Redis mutation path
- No raw prompts stored in `source_data`
- No `chain_of_thought` fields accepted or rendered
- No raw storage identifiers in API responses/UI/logs
- Worker endpoints internal-only (Docker network, not published to host)
